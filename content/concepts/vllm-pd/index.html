    vLLM PD 分离：把 prefill、decode 和 KVCache 传输拆开调度

    问题：长上下文请求里，prefill 很重、decode 要稳，混在同一批 GPU 上会互相干扰。
    方案：vLLM 用两个实例分别跑 prefill 和 decode，再用 KV connector 把 prefill 产出的 KVCache 交给 decode。
    产物：学完后你能分清 PD 分离、KV connector、NIXL/UCX、Mooncake、LMCache 分别解决哪一层问题。

    来源：vLLM latest 官方文档 Disaggregated Prefilling、NixlConnector Usage Guide、MooncakeConnector Usage Guide、vLLM GitHub main 源码、UCX 官方网站、Mooncake 官方文档。

    前置：最好知道 LLM 推理分 prefill 和 decode 两阶段，以及 KVCache 是 attention 层缓存下来的 key/value 张量；本文会重新解释。

    一句话：vLLM 的 PD 分离不是“多开几个服务”这么简单。真正的核心是：prefill 实例负责算出 KVCache，decode 实例负责继续生成 token，而中间的 KVCache 传输、匹配、等待、失败处理，都被抽象成 KV connector。

    术语速查

    本章你将知道：读 vLLM PD、UCX、Mooncake 文档前必须先懂的词。

    术语大白话解释
    prefill预填充阶段：模型一次性处理用户输入的整段 prompt，算出后续生成要用的 KVCache。长 prompt 下，这一步很重。
    decode解码阶段：模型一次生成一个新 token。每一步都要读取历史 KVCache，所以更在意稳定延迟和显存/带宽。
    PD 分离Prefill-Decode disaggregation，把 prefill 和 decode 拆到不同 vLLM 实例或不同资源池里运行。
    KVCacheKey-Value Cache，Transformer attention 层缓存的 key/value 张量。decode 复用它，就不用反复重算历史 token。
    KV connectorvLLM 里连接 prefill/decode 和外部传输系统的抽象。它决定 KVCache 怎么发、怎么收、什么时候算加载完成。
    NIXLNVIDIA Inference Xfer Library，vLLM 的高性能 KV 传输 connector 可用它做异步跨进程/跨节点传输。
    UCXUnified Communication X，一个面向高性能计算和数据中心应用的通信框架。它不是 vLLM connector，而是 NIXL 可使用的底层传输后端。
    MooncakeConnectorvLLM 的 Mooncake KV 传输 connector，面向 PD 分离，把 prefill 侧 KVCache 传给 decode 侧。
    MooncakeStoreConnectorvLLM 接入 Mooncake Store 的 connector，更偏“分布式 KVCache 存储/缓存池”，可用于单机 offload、PD 分离和磁盘 offload。
    LMCacheLLM serving 的 KVCache 管理系统。vLLM 有 LMCache connector，可用 LMCache 保存/复用 KVCache；文档示例里也会把 NIXL 当底层传输。

    一、问题：prefill 和 decode 的资源形状不同，混跑会互相拖累

    本章你将知道：为什么 vLLM 要把一个请求拆成两个实例协作，而不是只在一个 worker 里跑完。

    一次 LLM 推理可以粗略分成两段。用户刚把长 prompt 发进来时，模型要把整段输入读完，这叫 prefill。prefill 会产生 KVCache。之后模型开始一个 token 一个 token 地输出，这叫 decode。decode 每一步都要读之前的 KVCache。

    这两段的优化目标不一样。prefill 喜欢“大块吞吐”：输入越长，越需要把计算堆满。decode 喜欢“稳定小步”：每个 token 都要尽快出来，不能被一个很长的 prefill 卡住。普通部署把两段放在同一批实例里，简单但容易互相干扰。

    关键点：PD 分离本身只解决“谁来算 prefill、谁来算 decode”。它马上带来第二个问题：prefill 算出的 KVCache 不在 decode 实例上，所以必须有一条可靠、高效、可调度的 KVCache 传输路径。

    不分离：
请求 → 同一个 vLLM 实例
       ├─ prefill：算 prompt，生成 KVCache
       └─ decode：读本地 KVCache，逐 token 生成

PD 分离：
请求 → prefill 实例：算 prompt，生成 KVCache
          ↓ KV connector 负责传输/匹配/确认
       decode 实例：拿到 KVCache，逐 token 生成

    vLLM 官方文档明确把 Disaggregated Prefilling 标为 experimental（实验性功能），并说明当前实现方式是运行 2 个 vLLM instances：一个 prefill instance，一个 decode instance，然后用 connector 把 prefill KV caches 和结果从 prefill 传到 decode。

    二、方法：vLLM 把 KVCache 传输抽象成 scheduler connector 和 worker connector

    本章你将知道：vLLM PD 分离里的请求流、KV 流、源码入口，以及 UCX/Mooncake 站在哪一层。

    2.1 一条请求在 vLLM PD 里的最小路径

    从概念上看，一条请求会经历 5 步：decode 侧接到请求，发现需要远端 KV；prefill 侧处理 prompt；connector 把 KVCache 和必要元数据传过来；decode 等待相关层的 KV 加载完成；最后 decode 继续生成输出。

    客户端
  ↓
decode vLLM instance
  ├─ scheduler：判断请求需要远端 KV，构造 connector metadata
  ├─ worker：在 attention 前等待 KV load
  ↓
KV connector
  ├─ scheduler connector：安排哪些请求要发/收 KV
  └─ worker connector：真正执行 KVCache save/load/transfer
  ↑
prefill vLLM instance
  ├─ worker：跑 prefill，逐层 save KV
  └─ connector：把 KVCache 暴露给 decode 或主动发送

    官方 README 还给了三层抽象：KV pipe（像 FIFO 管道，传 tensor）、KV lookup buffer（按 token/key 查 KVCache，解决 prefill 和 decode 处理顺序不同的问题）、KV connector（把 pipe/buffer 接进 vLLM）。这解释了为什么 PD 不能只靠一个简单队列：高 QPS 下 prefill 可能按 A→B→C 处理，但 decode 可能先需要 C。

    2.3 KVConnectorBase_V1 的生命周期：先加载，再逐层等待，再保存

    vLLM 的 connector base class 把核心动作拆成几个钩子。decode 侧会启动 KV 加载，attention 层执行前等待对应 layer 的 KV 到位；prefill 侧会在 layer 计算后保存 KV；请求结束后 connector 报告哪些请求已经完成传输。

    展开：用生命周期理解 base.py 里的几个方法

    bind_connector_metadata：scheduler 产出的 connector metadata 绑定到 worker，告诉 worker 这一轮有哪些请求要处理。

    start_load_kv：decode 侧开始加载远端 KVCache，可以是异步的。

    wait_for_layer_load：某一层 attention 真要用 KV 之前，在这里等这一层的 KV 到位。

    save_kv_layer：prefill 侧每算完一层，把这一层 KVCache 保存或发出去。

    wait_for_save：forward 结束前确认保存动作完成。

    get_finished：告诉 scheduler 哪些请求的发送或接收已经结束，便于释放资源和推进状态。

    展开：一个简化后的 vLLM PD 伪代码 
    # decode instance
metadata = scheduler_connector.build_connector_meta(scheduler_output)
worker_connector.bind_connector_metadata(metadata)
worker_connector.start_load_kv(forward_context)

for layer in model.layers:
    worker_connector.wait_for_layer_load(layer.name)
    layer.forward_with_remote_kv_ready()

finished_sends, finished_recvs = worker_connector.get_finished()

# prefill instance
for layer in model.layers:
    kv = layer.forward_and_produce_kv(prompt_tokens)
    worker_connector.save_kv_layer(layer.name, kv)

worker_connector.wait_for_save()

    2.4 NIXL 和 UCX：UCX 是传输后端，不是 vLLM connector

    最容易混淆的是 UCX。vLLM 当前文档里你会看到 NixlConnector，也会看到 UCX，但它们不在同一层。NixlConnector 是 vLLM connector；UCX 是 NIXL 默认可用的底层 transport library。也就是说，vLLM 调 connector，connector 调 NIXL，NIXL 再通过 UCX 这类后端走网络或共享内存。

    vLLM scheduler/worker
  ↓
NixlConnector / NixlPullConnector / NixlPushConnector
  ↓
NIXL library
  ↓
UCX / LIBFABRIC / GDS 等 backend
  ↓
网卡、GPU、共享内存、存储设备

    vLLM NixlConnector 文档说，它提供 fully asynchronous send/receive，用 NIXL 做高效跨进程 KVCache transfer；文档还写明 NIXL 支持多个 transport backend，默认使用 UCX，并可以通过 kv_connector_extra_config.backends 选择不同后端。UCX 官方把自己定义为 open-source、production-grade communication framework，支持 InfiniBand、共享内存和 GPU 等环境。

    展开：NIXL + UCX 的配置长什么样 
    # 这是概念化配置片段，具体命令要以 vLLM 官方文档和你的部署脚本为准。
# prefiller
--kv-transfer-config '{
  "kv_connector": "NixlConnector",
  "kv_role": "kv_producer",
  "kv_load_failure_policy": "fail",
  "kv_connector_extra_config": {
    "backends": ["UCX"]
  }
}'

# decoder
--kv-transfer-config '{
  "kv_connector": "NixlConnector",
  "kv_role": "kv_consumer",
  "kv_load_failure_policy": "fail",
  "kv_connector_extra_config": {
    "backends": ["UCX"]
  }
}'

    版本注意：vLLM NIXL 文档说明 kv_role="kv_both" 对 NixlConnector 已 deprecated，应该给 prefill 实例设置 kv_producer，给 decode 实例设置 kv_consumer。

    2.5 Mooncake：一个 connector 名字背后有两类能力

    Mooncake 在 vLLM 里容易有两个层次。第一类是 MooncakeConnector：它更像点对点的 KV transfer connector，用 Mooncake 的传输能力把 prefill 侧 KVCache 送到 decode 侧。第二类是 MooncakeStoreConnector：它接入 Mooncake Store，更像分布式 KVCache 存储/缓存池，可以用于单机 KV offload、PD 分离和 disk offload。

    名字重点你可以怎么理解
    MooncakeConnectorKVCache transferprefill 算出 KV，decode 需要 KV，中间用 Mooncake 传输能力搬过去。
    MooncakeStoreConnectorKVCache store/cache poolKV 不只是点对点搬运，还可以进入一个外部 KVCache 存储/缓存系统。
    Mooncake Transfer Engine高性能数据传输Mooncake 生态里的传输引擎，支持 RDMA 等能力，解决“大块 KV 怎么快搬”。
    Mooncake Store分布式 KVCache 存储Mooncake 生态里的 KVCache 存储后端，解决“KV 放哪里、怎么复用、怎么扩容”。

    源码上，vLLM 在 factory.py 同时注册了 MooncakeConnector 和 MooncakeStoreConnector。MooncakeConnector 源码里有 scheduler 和 worker 两个类；worker 会根据 kv_role 判断自己是 producer 还是 consumer，并注册 KV cache tensors 后执行发送/接收。

    2.6 LMCache、FlexKV、Offloading、MultiConnector：不要和 UCX/Mooncake 混成一层

    vLLM latest 文档列出的 connector 已经不止一种。它们有的偏“外部 KVCache 管理系统”，有的偏“传输库”，有的偏“本机 offload”，还有 MultiConnector 这种组合器。读源码或 PR 时，先问一句：这个东西是在解决“怎么传”，还是“放哪里”，还是“多个 connector 怎么编排”？

    方案主要解决和 PD 的关系
    NixlConnector跨进程/跨节点异步 KV 传输PD 分离里的通用高性能传输 connector，可选 UCX/LIBFABRIC/GDS 等 backend。
    UCX底层通信 transport不是 vLLM connector；常作为 NIXL 的默认传输后端。
    MooncakeConnector基于 Mooncake 的 KV 传输让 prefill/decode 实例通过 Mooncake 能力搬 KV。
    MooncakeStoreConnector外部分布式 KVCache 存储/缓存池适合把 KVCache 管理从 vLLM 实例里进一步外置。
    LMCacheConnectorV1接入 LMCache 做 KV 管理可用于 disaggregated prefill；官方文档示例提到它使用 NIXL 作为底层 KV transmission。
    FlexKVConnectorV1接入 FlexKV 分布式 KV store 和多级缓存更偏超大规模推理里的 KV 存储和多层缓存管理。
    OffloadingConnector把 KV offload 到 CPU 内存主要缓解本机 GPU 显存压力，不等价于跨节点 PD。
    MultiConnector按顺序组合多个 connector适合同时启用多种 KV transfer/offload 策略，但调试复杂度更高。

    三、总结：先分清层级，再选择 connector

    本章你将知道：怎样用一张决策表理解 vLLM PD、NIXL/UCX、Mooncake、LMCache 的关系。

    vLLM PD 分离可以按四层理解。第一层是 serving 架构：prefill 和 decode 分成两个 vLLM 实例。第二层是 vLLM 抽象：scheduler connector 负责调度 KV 传输，worker connector 负责执行 KV 保存/加载。第三层是 connector 实现：NIXL、Mooncake、LMCache、FlexKV 等。第四层是底层传输或存储：UCX、LIBFABRIC、GDS、Mooncake Store、LMCache server 等。

    我想找源码入口factory.py + base.py + 具体 connector 目录注册表告诉你 connector 名称映射到哪个类；base class 告诉你生命周期。
    我想用高性能点对点传 KVNixlConnectorNIXL 提供异步 send/recv，UCX 是它常用的底层传输后端。
    我想理解 UCXUCX 官方文档 + NIXL usage guideUCX 是通信层，不是 vLLM connector；不要在 vLLM 里找 UCXConnector。
    我想接 MooncakeMooncakeConnector / MooncakeStoreConnector 文档和源码先判断你要的是点对点传输，还是外部 KVCache 存储/缓存池。

    一句话回顾：vLLM PD 的主线是“两个实例 + 一个 KV connector 抽象”；UCX 是 NIXL 可用的底层通信后端；Mooncake 既可以作为 KV 传输能力，也可以作为 KVCache 存储/缓存池接入。
