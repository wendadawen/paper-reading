    Mooncake：把 KVCache 当成推理系统的一等公民

    问题：长上下文 LLM 推理里，瓶颈不只是算 token，还包括 KVCache 放在哪里、怎么搬、什么时候复用。
    方案：Mooncake 把 prefill 和 decode 拆成不同集群，并围绕 KVCache 做缓存、调度、传输和过载拒绝。
    产物：学完后你能看懂 Mooncake 为什么适合长上下文服务，以及它和普通连续批处理推理系统的关键差别。

    来源：论文 Mooncake: A KVCache-centric Disaggregated Architecture for LLM Serving、官方文档 Welcome to Mooncake、官方仓库 kvcache-ai/Mooncake。

    前置：最好先知道 Transformer 推理、attention、KVCache、prefill/decode 两阶段；本文会在第一次出现时重新解释。

    一句话：Mooncake 是一个面向大规模 LLM serving（大语言模型在线推理服务）的系统。它的核心观点是：长上下文请求里，KVCache（注意力层缓存下来的 key/value 张量）本身已经大到值得被当成一种独立资源来管理，而不是 GPU 显存里的附属品。

    术语速查

    本章你将知道：读 Mooncake 时最容易卡住的几个词是什么意思。

    术语大白话解释
    LLM serving把大语言模型部署成线上服务，持续接收用户请求并返回生成结果。重点不是训练模型，而是如何低延迟、高吞吐地推理。
    KVCacheKey-Value Cache，Transformer attention 层缓存下来的 key/value 张量。生成下一个 token 时可以复用历史 token 的 key/value，避免重复计算整段历史。
    prefill预填充阶段：模型一次性处理用户输入的整段 prompt，并产生后续 decode 要用的 KVCache。长 prompt 下，这一步计算量很大。
    decode解码阶段：模型一次生成一个新 token。每一步通常只处理最新 token，但要读取历史 KVCache，所以更容易受显存带宽和调度延迟影响。
    PD disaggregationPrefill-Decode disaggregation，把 prefill 和 decode 放到不同机器或不同 GPU 池里跑，让两种负载独立扩缩容和调度。
    SLOService Level Objective，服务质量目标。例如首 token 延迟不能超过某个阈值，或相邻 token 间隔不能太长。
    TTFTTime To First Token，从请求进入系统到返回第一个 token 的时间。用户体感里，“它多久开始回复”主要看这个。
    TBTTime Between Tokens，相邻两个输出 token 的间隔。用户体感里，“回复是不是流畅”主要看这个。
    Mooncake Store官方开源的分布式 KV cache 存储引擎，用来在多实例、多层存储之间保存和共享 KVCache。
    Transfer EngineMooncake 的高性能数据传输组件，给 KVCache、权重或张量搬运提供统一接口，底层可使用 TCP、RDMA、NVLink 等传输方式。

    一、问题：长上下文推理的瓶颈不只在算力，还在 KVCache 管理

    本章你将知道：为什么长 prompt 会把普通 LLM serving 系统推到一个尴尬位置。

    一个 LLM 请求通常分成两段。第一段是 prefill：把用户输入的 prompt 全部读进去。第二段是 decode：一个 token 一个 token 地往外生成。短上下文时，这两个阶段可以放在同一套 GPU worker 里混着跑；长上下文时，它们的资源需求会变得很不一样。

    prefill 更像“大块计算”：prompt 越长，要处理的 token 越多，attention 计算和中间张量都很重。decode 更像“持续小步生成”：每一步只来一个新 token，但必须反复读取历史 KVCache。如果系统把两者绑在一起调度，就会出现两个问题：prefill 大请求会拖慢 decode 的流畅输出，decode 的低延迟需求又会限制 prefill 的批量效率。

    关键点：Mooncake 不是只说“把 prefill 和 decode 拆开”。它更进一步说：拆开之后，真正要围绕 KVCache 重新设计调度、缓存和传输，否则只是把问题从 GPU 计算挪到了网络和内存搬运。

    KVCache 为什么值得单独管理？因为它既大，又有复用价值。长上下文请求会产生大量 KVCache；很多真实业务请求还会共享前缀，例如系统 prompt、长文档、代码仓库上下文、对话历史的一部分。如果能复用已有 KVCache，就可以少做 prefill；如果复用失败，系统就要重新计算和存储一大段历史。

    普通系统的朴素做法

    朴素做法是：请求到某个 GPU worker 后，prefill 和 decode 都在这个 worker 或同一组 worker 上完成，KVCache 主要存在本地 GPU 显存里。这个做法简单，但它隐含了一个假设：KVCache 跟着请求走，本地管理就够了。

    请求进入
  ↓
同一组 GPU worker
  ├─ prefill：读完整 prompt，生成 KVCache
  └─ decode：反复读本地 KVCache，逐 token 输出

    Mooncake 面向的是 Kimi 这类真实长上下文服务。论文摘要明确说，它是 Moonshot AI 的 Kimi serving platform，并在真实工作负载下让 Kimi 能处理更多请求。这个场景下，KVCache 不再只是“某个请求的临时中间结果”，而是影响系统吞吐、延迟和拒绝策略的核心资源。

    二、方法：把 prefill、decode、KVCache 池和调度器拆开协同

    本章你将知道：Mooncake 如何把 KVCache 从 GPU 本地缓存变成可调度、可复用、可传输的系统资源。

    直觉：先按工作类型拆机器，再按 KVCache 位置调度请求

    Mooncake 的第一层设计是 prefill/decode 分离：prefill cluster（专门跑预填充的 GPU 集群）负责处理长 prompt，decode cluster（专门跑逐 token 生成的 GPU 集群）负责持续输出。这样两类负载可以独立优化：prefill 更关注大批量计算效率，decode 更关注低延迟和稳定 token 间隔。

    第二层设计是 disaggregated KVCache pool（分离式 KVCache 池）：利用 GPU 集群里相对空闲的 CPU、DRAM 和 SSD，把 KVCache 放到 GPU 显存之外的多级存储里。这样 KVCache 可以跨请求、跨实例复用，也可以在 prefill 和 decode 之间传输。

    用户请求
  ↓
KVCache-centric scheduler
  ├─ 查前缀：哪些 token 的 KVCache 已经缓存？
  ├─ 估延迟：复用缓存节省多少 prefill？搬运缓存要花多少时间？
  └─ 选实例：把请求派到综合代价最低的位置
        ↓
  prefill cluster  ── 生成/补齐 KVCache ──→  KVCache pool
        ↓                                      ↓
  decode cluster   ←────── 读取/接收 KVCache ─┘
        ↓
  逐 token 输出

    核心机制 1：KVCache-centric scheduler 不只看负载，还看缓存命中

    普通负载均衡可能只看“哪台机器空”。Mooncake 的调度器还要看“哪台机器或缓存池里已经有这个请求可复用的 KVCache”。如果某个请求的长前缀已经缓存，调度器可以把它路由到更容易复用这些缓存的位置，减少重复 prefill。

    可以把调度目标理解成下面这个解释性打分式。注意：这不是论文逐字给出的正式公式，而是把官方描述的“缓存命中、排队、传输、执行时间”抽象成一个便于理解的模型。

    $$ score(i) = queue(i) + prefill(unmatched\_tokens, i) + transfer(remote\_cache, i) $$

    展开：这个调度打分式在表达什么

    目标：对每个候选实例 $i$，估计请求如果交给它处理，首 token 延迟大概会是多少。

    第 1 项 $queue(i)$：实例 $i$ 前面已经排了多少活。机器再适合复用缓存，如果队列很长，也可能让 TTFT 变差。

    第 2 项 $prefill(unmatched\_tokens, i)$：请求里没有命中缓存的 token 还需要重新 prefill。命中的前缀越长，这一项越小。

    第 3 项 $transfer(remote\_cache, i)$：如果缓存不在目标实例本地，就要把 KVCache 从远端搬过来。复用缓存不是免费的，网络和内存拷贝也会增加延迟。

    最终选择：调度器倾向选择综合代价最低的实例，而不是单纯选择最空闲或缓存最多的实例。

    核心机制 2：分离式 KVCache 池把 CPU、DRAM、SSD 也纳入推理资源

    官方文档和论文都强调，Mooncake 会利用 GPU 集群中未充分使用的 CPU、DRAM 和 SSD 资源来构建 KVCache pool。这里的重点不是“把东西全丢到磁盘”，而是多层缓存：热数据尽量靠近 GPU，较冷但可能复用的数据可以留在 CPU 内存或 SSD 中。

    层级适合放什么代价
    GPU 显存当前正在 decode 的活跃请求 KVCache最快，但最贵、容量最紧张
    CPU DRAM近期可能复用的前缀 KVCache容量更大，但需要 CPU/GPU 或跨机传输
    SSD/NVMe更冷、但仍可能被命中的大块 KVCache容量最大，但读取延迟更高

    Mooncake Store 是这个方向的开源组件之一。官方仓库描述它是分布式 KV cache storage engine，支持大对象并行 I/O、多级缓存和独立于推理引擎的弹性存储。对小白来说，可以先把它理解成“专门给 LLM KVCache 用的分布式缓存后端”。

    核心机制 3：Transfer Engine 专门负责把大块张量高效搬走

    只要 prefill 和 decode 分离，KVCache 就必须在节点之间移动。Transfer Engine 是 Mooncake 的传输组件，官方仓库列出的能力包括统一数据移动接口、多种传输协议、拓扑感知路径选择和多网卡带宽聚合。这里不用先记所有协议名，只要抓住一点：Mooncake 把“搬 KVCache”当成系统里的显式问题，而不是让框架默认通信顺手解决。

    展开：用伪代码看 Mooncake 风格的请求调度 
    def schedule_request(request, prefill_nodes, cache_pool):
    # request.prompt_tokens: 用户输入 token
    # cache_pool: 分布式 KVCache 池，能查某段前缀是否已有缓存
    prefix_hit = cache_pool.find_longest_prefix(request.prompt_tokens)

    best_node = None
    best_score = float("inf")

    for node in prefill_nodes:
        queue_cost = estimate_queue_delay(node)
        prefill_cost = estimate_prefill_time(
            unmatched_tokens=len(request.prompt_tokens) - prefix_hit.length,
            node=node,
        )
        transfer_cost = estimate_transfer_time(
            cache_location=prefix_hit.location,
            target_node=node,
        )
        score = queue_cost + prefill_cost + transfer_cost

        if score < best_score:
            best_score = score
            best_node = node

    return best_node, prefix_hit

    过载场景：有些请求要提前拒绝，避免浪费 prefill

    论文特别强调，真实线上服务会遇到严重过载，不是所有请求都能被处理。一个危险做法是：先让请求完成昂贵的 prefill，后来发现 decode 阶段已经无法满足 SLO，再拒绝或超时。这会浪费大量 GPU 计算。

    Mooncake 的思路是 prediction-based early rejection（基于预测的提前拒绝）：在请求进入昂贵的计算阶段前，先预测它未来是否还能满足延迟目标。如果系统判断它大概率来不及，就早点拒绝，避免占用 prefill 资源。

    容易误解：提前拒绝不是为了“少服务用户”，而是在过载时保护有效吞吐。已经注定违反 SLO 的请求继续跑，会拖累本来可以按时完成的请求。

    三、总结：Mooncake 把推理优化从算子层推到系统资源层

    本章你将知道：Mooncake 的本质、适用场景，以及它和相关方案的区别。

    Mooncake 的本质不是某一个 kernel 优化，也不是单纯加一个远端缓存。它把长上下文 LLM serving 拆成四个协同问题：prefill 怎么批量处理，decode 怎么稳定输出，KVCache 怎么跨层级保存和复用，调度器怎么在命中率、排队时间、传输成本和 SLO 之间做取舍。

    方案核心思路优点代价
    同实例 prefill+decode一个 worker 或同一组 GPU 同时处理输入和生成结构简单，KVCache 多数留在本地长上下文下 prefill 和 decode 互相干扰，资源难以独立优化
    普通 PD 分离prefill 和 decode 分到不同节点两类负载可以独立扩缩容KVCache 传输和缓存复用会成为新瓶颈
    MooncakePD 分离 + KVCache 池 + cache-aware scheduler + Transfer Engine更适合长上下文、前缀复用和过载服务场景系统复杂度更高，需要管理缓存、传输、调度和 SLO 预测
    单机 prefix cache只在本地 worker 复用相同前缀容易接入，适合小规模重复前缀跨实例复用弱，缓存容量受单机限制

    什么时候适合关注 Mooncake

    • 你的服务有大量长上下文请求，例如长文档问答、代码仓库问答、复杂 agent 上下文。
    • 请求之间存在可复用前缀，例如固定 system prompt、共享文档、共享会话历史。
    • GPU 集群里 prefill 和 decode 的资源需求差异明显，混跑导致延迟或吞吐不稳定。
    • 你已经在使用或评估 vLLM、SGLang、TensorRT-LLM、LMCache 等推理生态，并需要远端 KVCache 或 PD 分离能力。

    什么时候不要急着用它

    • 你的请求大多很短，prefill 成本不高，KVCache 复用空间也小。
    • 部署规模很小，系统复杂度比节省的 GPU 成本更重要。
    • 团队还没有能力定位跨机传输、缓存一致性、延迟预测和过载调度问题。
    • 你只是在做离线实验，尚未遇到线上 SLO 和峰值过载问题。

    一句话回顾：Mooncake 的贡献在于把“KVCache 在哪里、怎么复用、怎么搬、什么时候值得搬”变成 LLM serving 的中心调度问题。
