# 概念学习 workflow

这个 workflow 只服务一个目标：让小白在足够上下文里真正看懂一个概念。

它不按 algorithm / system / map 预先分类。很多概念天然混合：既有流程，也有公式，也有工程实现。硬分类型会增加维护成本，也会让作者为了套模板而牺牲理解路径。

---

# 一、总原则

## 1.1 先确认读者目标

每篇 concept 开始前，先问读者 5 个定位问题：

1. 你想看懂什么？
2. 你为什么要学它？
3. 你现在已知什么？
4. 你现在卡在哪里？
5. 学完后你要能回答或判断什么？

默认读者是完全小白。即使读者自称懂一部分，正文仍要补足理解本文必需的上下文。

## 1.2 上下文合格标准

上下文不是术语表。合格标准是：在正式讲机制前，读者已经能预测新概念的大致方向。

也就是说，读者应该已经知道：

- 旧方案哪一步走不下去
- 新概念大概率要替换、拆分、缓存、调度、约束、估计或传输哪一步
- 如果没有这个概念，代价会在哪里爆炸

## 1.3 研究和写作必须分开

先研究清楚，再写 HTML。

研究阶段必须产出“研究+教学卡片”。卡片不进入最终页面，但最终页面的 context-box、读者目标、误区和正文路径都必须来自这张卡片。

卡片可以很短，但不能跳过。

# 二、研究阶段：研究+教学卡片

写 HTML 前，先填下面这张卡片。

```markdown
## A. 读者和边界

### 1. 读者目标

读者到底想看懂什么？

### 2. 概念边界

这个概念指什么？不指什么？有没有同名概念、相邻概念或容易混淆的说法？

### 3. 权威来源

- 官方文档：
- 论文 / 技术报告：
- 官方源码 / 配置 / 设计文档：
- 实测或示例：
- 第三方资料（只辅助理解，不作为核心事实）：

## B. 理解路径

### 4. 大流程定位

它属于哪个大流程、哪一层？上游是什么？下游是什么？

### 5. 旧世界

没有它时，人们怎么做？系统、算法或流程怎么运转？

### 6. 断点

旧方式哪里走不下去？具体是慢、贵、不稳、不准、难扩展、难维护，还是目标不对？

### 7. 新想法

如果自己设计，会自然想到什么方向？拆分、缓存、调度、加约束、换目标、加 baseline、换通信方式，还是别的？

### 8. 正式定义

这个概念的技术定义是什么？一句话小白定义是什么？正式定义是什么？

## C. 写作落点

### 9. 机制草图

它如何工作？主路径用什么讲最清楚：公式、数据流、伪代码、源码路径、配置、最小请求，还是最小数值例子？

### 10. 边界对比

什么时候用？不解决什么？和哪些相邻概念区分？

### 11. 证据清单

- 官方文档明确说了什么？
- 论文 / 技术报告明确说了什么？
- 源码 / 配置明确证明了什么？
- 哪些是本文为了教学做的解读者推断？
```

如果卡片里的“旧世界”“断点”“新想法”写不出来，不要进入写作阶段。

# 三、写作阶段：页面结构

使用模板：

```bash
mkdir -p content/concepts/<concept-name>
cp templates/concept/index.html content/concepts/<concept-name>/index.html
```

最终页面按下面顺序写。

## 3.1 顶部模块

顶部固定包含：

1. title
2. reading time
3. callout：问题 / 方案 / 产物
4. context-box：大流程 / 旧方案 / 痛点 / 改变点
5. 读完你能回答（3-5 个问题）
6. 最容易误解（2-4 个误区）
7. meta：来源 / 前置
8. 小白版一句话定义

顶部要让读者 30 秒内知道：这个概念是什么、为什么要学、站在哪里、最容易误解什么。

## 3.2 正文 H1 结构

正文固定使用这条理解路径：

```text
一、定位：它在整个流程里的哪一步
二、旧世界：没有它时会怎么做
三、断点：为什么旧办法自然走不下去
四、新想法：这个概念改变了哪一步
五、正式定义：它到底是什么
六、机制：它怎么工作
七、边界：什么时候用，什么时候不用
证据边界声明
```

不要写“本章你将知道”。每个 H1 后用 1-2 句自然导语说明这一章要推进哪一步理解。

# 四、正文写法

## 4.1 一、定位

回答：它在整个流程里的哪一步？

必须写：

- 大流程图或文字流程
- 当前概念所在位置
- 上游输入是什么
- 下游影响是什么

## 4.2 二、旧世界

回答：没有它时会怎么做？

必须写：

- 旧流程
- 旧方案为什么看起来合理
- 旧方案依赖什么假设

## 4.3 三、断点

回答：为什么旧办法自然走不下去？

必须写具体断点，不写空泛形容词。

坏在哪里要落到：

- 计算成本
- 显存 / 内存 / 网络
- 样本效率
- 稳定性
- 正确性
- 维护复杂度
- 目标不一致
- 扩展性

## 4.4 四、新想法

回答：这个概念改变了哪一步？

这一章不是正式定义，而是自然引出。读者应该感觉：如果我自己设计，也会朝这个方向走。

章末必须有检查点列表。用普通 bullet，不用 callout：

```text
读到这里，你应该已经能解释：
- 旧方案卡在哪一步
- 新概念为什么自然会改这一点
- 如果不改，代价会在哪里继续变大
```

## 4.5 五、正式定义

回答：它到底是什么？

这里给两层定义：

1. 小白版一句话定义：不用术语或少用术语
2. 正式技术定义：可以使用术语，但首次出现必须解释

## 4.6 六、机制

机制用三层展开：

1. **直觉层**：一句话讲它在做什么
2. **流程层**：步骤 / 数据流 / 变量流 / 状态流
3. **细节层**：公式 / 源码 / 配置 / 伪代码，放折叠块

如果有最小例子，放折叠块。不需要在主文预告。

适合折叠：

- 最小数值例子
- 最小请求 / 数据流
- 最小调用链
- 完整公式推导
- 伪代码 / 可运行代码
- 源码路径和配置片段

## 4.7 七、边界

回答：什么时候用，什么时候不用？

必须写：

- 它解决什么
- 它不解决什么
- 和相邻概念怎么区分
- 什么时候该用
- 什么时候不该用

# 五、证据边界声明

证据集中写在最后，按来源分：

```text
官方文档
  - 明确支持哪些事实

论文 / 技术报告
  - 明确支持哪些机制、公式、实验结论

官方源码 / 配置
  - 明确支持哪些实现细节、路径、参数名

实测 / 示例
  - 明确支持哪些运行现象

解读者推断
  - 本文为了教学做了哪些类比、简化、流程抽象或判断
```

正文不要密集插入引用，除非不标会造成误解。

---

# 六、折叠块规则

主路径放理解必须的信息；折叠块放细节。

不许只放在折叠块：

- 关键假设
- 核心直觉
- 会改变结论的限制条件
- 旧方案断点
- 新概念改变点

# 七、发布步骤

在 `content.json` 的 `items` 增加条目：

```json
{
  "id": "concept-name",
  "section": "concepts",
  "tag": "concept-name",
  "title": "概念名：一句话概括",
  "desc": "读者点进来能学到什么",
  "path": "content/concepts/concept-name/"
}
```

运行：

```bash
python scripts/generate_index.py
python scripts/validate_content.py
```

# 八、核查清单

## 8.1 研究阶段

- [ ] 已完成研究+教学卡片
- [ ] 权威来源以官方文档 / 论文 / 源码 / 实测为主
- [ ] 概念边界清楚，没有混同同名概念或相邻概念
- [ ] 旧世界、断点、新想法能连成自然链条

## 8.2 写作阶段

- [ ] 顶部包含 callout、context-box、读者目标、常见误区、meta、小白定义
- [ ] “读完你能回答”有 3-5 个问题
- [ ] “最容易误解”有 2-4 个误区
- [ ] 正文 H1 使用固定理解路径
- [ ] 每个 H1 后有自然短导语
- [ ] “四、新想法”末尾有检查点列表
- [ ] “六、机制”按直觉层 / 流程层 / 细节层展开
- [ ] 最小例子如有需要，放折叠块
- [ ] 不写“本章你将知道”

## 8.3 质量门禁

- [ ] 读者在正式定义前，已经能预测新概念方向
- [ ] 关键假设没有藏在折叠块
- [ ] 术语解释服务推理，不是词典
- [ ] 证据边界按来源分
- [ ] 事实和解读者推断分开
- [ ] 没有凭印象写当前源码、配置或性能数字
- [ ] `python scripts/validate_content.py` 通过
