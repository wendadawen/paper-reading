# 概念教学文档写作流程

**产物**：纯 HTML + KaTeX（本地库）。每个概念一个目录，主入口 `content/concepts/<concept-name>/index.html`。

**读者画像**：零基础入门训练/RL/post-training/推理系统的工程师。有技术基础但不懂这个领域。30 秒能抓要点，展开折叠能看到完整推导、伪代码或系统流程。

**模板**：`templates/concept/index.html`。拷贝改占位符即可，不要从空白 HTML 写起。

**首页**：新增内容后改 `content.json`，再运行 `python scripts/generate_index.py`。

---

# 一、流程

### 1. 确认概念范围

回答三个问题，答案贯穿全文：

1. **一句话本质**：这个概念用 X 解决了 Y 问题
2. **痛点是什么**：为什么需要这个概念（前置概念的什么局限）
3. **核心机制**：必须讲清的是公式、算法流程、系统架构，还是工程 trade-off

如果是工程系统类概念，不要硬写公式；用数据流、状态转换、组件职责和伪代码替代。

### 2. 收集证据

- 优先官方文档、论文、官方源码、官方博客
- 每个关键论断记录出处
- 源码路径必须查当前版本，不凭记忆写
- 找不到出处就放进“解读者推断”，不要混成事实

### 3. 拷贝模板

```bash
mkdir -p content/concepts/<concept-name>
cp templates/concept/index.html content/concepts/<concept-name>/index.html
```

改占位符：`<title>`、`<h1 class="title">`、`nav-brand`（写简短概念名，如 `RLHF`）、`blockquote.meta` 里的来源和前置。

### 4. 写内容

按“规范”部分的结构写。复杂公式、伪代码、源码片段和工程流程放 `<details>` 折叠；主路径只讲直觉、关键结论和读者必须知道的假设。

### 5. 更新首页

在 `content.json` 的 `items` 增加一条：

```json
{
  "id": "concept-name",
  "section": "concepts",
  "tag": "concept-name",
  "title": "概念名：一句话概括",
  "desc": "读者点进来能学到什么",
  "path": "content/concepts/concept-name/"
}
```

运行：

```bash
python scripts/generate_index.py
python scripts/validate_content.py
```

# 二、规范

## 主文档结构

严格按此顺序，H1 用结论式标题：

```text
顶部 callout（30 秒层）
  问题 → 方案 → 产物，各一句话。不写泛泛效果。

blockquote.meta
  来源 + 前置。

一句话本质段
  把 callout 展开成 2-3 句。

术语速查
  只列正文反复出现的术语。不能用未解释的术语解释术语。

一、问题（H1，结论式标题）
  本章你将知道：<一句话>
  先讲痛点，再引出这个概念要解决什么。

二、方法（H1，结论式标题）
  本章你将知道：<一句话>
  先讲直觉，再讲公式 / 算法 / 架构 / 数据流。
  复杂内容放 details 折叠。

三、总结 / 对比（H1，结论式标题）
  本章你将知道：<一句话>
  方法本质回顾 + 和相关概念的对比表格。

证据边界声明（H1）
  本章你将知道：每个论断的出处。
  分两段：事实/公式出处 + 解读者推断。
```

每个 H1（含“术语速查”和“证据边界声明”）下面都要有“本章你将知道”。

## 写作要点

- 顶部 callout 只写“问题 / 方案 / 产物”
- H1 串起来要能看出概念故事线
- 关键假设用 callout 在主路径标注，不藏折叠块
- 类比克制使用：1-2 句建立直觉，之后回到技术语言
- 数值例子用真实场景，不用抽象 token
- 证据出处集中在“证据边界声明”
- 不单独维护和 workflow 重复的 prompt；AI 生成辅助规则直接沉淀到本文件

## 公式、代码和工程流程

- 数学概念：正文给直觉和最终式，完整推导放 `<details>`
- 工程概念：正文给主流程，源码路径 / 伪代码 / 配置片段放 `<details class="code-details">`
- 代码块用 Prism.js：`<pre><code class="language-python">...</code></pre>`
- 不要把 `$...$` 包在 `<code>` 里，KaTeX 不扫描 `<code>` 内部

# 三、核查清单

### 外行友好度

- [ ] 只读 callout + H1 + “本章你将知道”，能不能复述这个概念做了什么？
- [ ] 术语表有没有用未解释术语解释术语？
- [ ] 方法章节第一段是不是直觉？
- [ ] 最技术的一句话，外行能不能读懂？

### 事实准确

- [ ] 每个关键论断有出处，集中在“证据边界声明”
- [ ] 事实和解读者推断分开
- [ ] 源码路径、配置名、性能数字都查过当前来源
- [ ] 不确定的信息标“未验证”或删除

### 格式

- [ ] 每个 H1 都有“本章你将知道”
- [ ] 复杂公式 / 代码 / 流程放折叠块
- [ ] 宽表格用 `.table-scroll`
- [ ] 没有模板占位符残留
- [ ] 本地资源路径指向 `../../../libs/`
- [ ] `python scripts/validate_content.py` 通过

# 四、铁律

1. **证据优先**：找不到出处就删，或标为解读者推断。
2. **直觉先行**：先讲为什么，再讲怎么做。
3. **关键假设不藏折叠块**：主路径必须看到。
4. **不残留模板占位符**：title、nav、meta、术语表都要改完。
5. **首页 manifest 驱动**：不要手改首页卡片，改 `content.json` 后生成。
